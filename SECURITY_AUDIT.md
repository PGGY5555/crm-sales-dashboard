# CRM 系統資安審查報告

**審查日期**：2026-03-13  
**審查範圍**：sales_dashboard 全系統（前端、後端 API、資料庫互動）  
**審查人**：Manus AI

---

## 一、審查總結

本次審查針對「CRM 銷售分析儀表板」系統進行全面的安全性檢視，涵蓋前端敏感資訊暴露、後端 API 認證授權、輸入驗證、個資保護、SQL 注入防護等五大面向。整體而言，**系統的安全基礎架構是合格的**，並非那種「把資料庫帳密寫在前端」或「API 完全不設防」的情況。但仍有若干可以強化的地方，以下逐項說明。

---

## 二、各項審查結果

### 2.1 前端是否暴露敏感資訊

| 檢查項目 | 結果 | 說明 |
|----------|------|------|
| 資料庫連線字串（DATABASE_URL） | **安全** | 前端程式碼中完全沒有出現 DATABASE_URL，連線字串僅存在於後端環境變數 |
| JWT_SECRET | **安全** | 僅在後端使用，前端無任何引用 |
| BUILT_IN_FORGE_API_KEY（後端密鑰） | **安全** | 前端無引用，僅後端使用 |
| 前端環境變數（VITE_ 開頭） | **安全** | 僅暴露 VITE_OAUTH_PORTAL_URL、VITE_APP_ID、VITE_FRONTEND_FORGE_API_KEY 等設計上就是給前端使用的公開值 |
| API Token（Shopnex） | **安全** | 使用 AES-256-CBC 加密存入資料庫，前端僅顯示遮罩值（前4後4字元） |

**結論：前端沒有暴露任何敏感的後端密鑰或資料庫連線資訊。** 這一點做得很好。

---

### 2.2 後端 API 認證與授權機制

| 檢查項目 | 結果 | 說明 |
|----------|------|------|
| tRPC 路由認證 | **安全** | 所有業務 API 均使用 `protectedProcedure`，要求有效的登入 session |
| Express 路由認證 | **安全** | 所有 `/api/upload/*` 和 `/api/import/*` 路由都呼叫 `verifyAuthSession()` 驗證 |
| 管理員功能保護 | **安全** | 使用者管理、權限管理、操作日誌等均檢查 `ctx.user.role !== "admin"` |
| 細粒度權限系統 | **安全** | 數據同步、Excel 匯入、API 憑證等操作均透過 `checkUserPermission()` 檢查 |
| Cookie 安全設定 | **安全** | 使用 httpOnly、secure、sameSite="none" |
| 未登入使用者 | **安全** | `requireUser` 中間件會擋下未認證的請求，回傳 UNAUTHORIZED |

**結論：認證與授權機制完整，沒有「裸露」的 API。** 每個 API 端點都需要登入，敏感操作還需要額外的角色或權限檢查。

---

### 2.3 輸入驗證與大量資料注入防護

| 檢查項目 | 結果 | 風險等級 | 說明 |
|----------|------|----------|------|
| tRPC 輸入驗證（Zod） | **安全** | — | 所有 tRPC 端點都使用 Zod schema 驗證輸入型別和格式 |
| 批次操作上限 | **安全** | — | 批次刪除/更新有 5,000 筆安全上限 |
| 檔案上傳大小限制 | **安全** | — | multer 限制 50MB，express.json 限制 50MB |
| list API 的 limit 參數 | **需改善** | 中 | `limit` 參數使用 `z.number()` 但**沒有設定 `.max()` 上限**，攻擊者可傳入 `limit: 999999` 一次拉出全部資料 |
| export API 無數量限制 | **需改善** | 中 | `getCustomerManagementExport` 直接用 `limit: 100000`，大量資料匯出可能造成記憶體壓力 |
| batch import 無批次大小限制 | **需改善** | 中 | `/api/import/batch` 接收前端傳來的 JSON 陣列，但**沒有檢查 batch 陣列長度**，理論上可以一次送入超大量資料 |
| Rate Limiting | **需改善** | 中 | **完全沒有 rate limiting 機制**，攻擊者可以高頻呼叫 API |

**結論：基本的型別驗證做得不錯（Zod），但缺少數量上限和頻率限制。**

---

### 2.4 個資相關 API 的資料暴露風險

| 檢查項目 | 結果 | 風險等級 | 說明 |
|----------|------|----------|------|
| 客戶列表 API | **需改善** | 中 | 使用 `db.select()` 返回**所有欄位**，包含 `rawData`（原始 Excel 匯入資料的完整 JSON）。F12 Network 可看到每筆客戶的完整個資 |
| 客戶詳情 API | **需改善** | 中 | 同上，返回完整客戶物件包含 rawData |
| 訂單列表/詳情 API | **需改善** | 中 | 同上，返回完整訂單物件包含 rawData |
| 匯出 API | **可接受** | 低 | 匯出功能需要完整資料，但需要登入才能使用 |
| 使用者列表 API | **安全** | — | 只選取必要欄位（id, name, email, role 等），不返回敏感資訊 |
| API Token | **安全** | — | 加密儲存，前端只看到遮罩值 |

**結論：最主要的問題是客戶和訂單 API 返回了 `rawData` 欄位。** 這個欄位包含原始 Excel 匯入的完整 JSON 資料，在 F12 Network 中可以看到。雖然需要登入才能存取，但對於已登入的一般使用者來說，這暴露了不必要的資訊。

### 2.5 SQL 注入與其他常見漏洞

| 檢查項目 | 結果 | 風險等級 | 說明 |
|----------|------|----------|------|
| Drizzle ORM 參數化查詢 | **安全** | — | 大部分查詢使用 Drizzle ORM 的 `sql` 模板字串，會自動參數化 |
| Excel 匯入的 esc() 函數 | **基本安全** | 低 | 手動跳脫單引號和反斜線，雖然不如參數化查詢理想，但能防止基本的 SQL 注入 |
| 客戶註冊趨勢的 lifecycle 篩選 | **有風險** | **高** | `getCustomerRegistrationTrend` 函數中，`filters.lifecycles` 直接拼接到 `sql.raw()` 字串中：`filters.lifecycles.map(l => \`'${l}'\`).join(',')` **沒有任何跳脫或驗證**，存在 SQL 注入風險 |
| 安全 HTTP Headers | **需改善** | 中 | 沒有使用 helmet 或手動設定安全 headers（X-Frame-Options、Content-Security-Policy 等） |
| XSS 防護 | **基本安全** | 低 | React 預設會跳脫 HTML 輸出，但沒有額外的 CSP header |

**結論：發現一處 SQL 注入風險（lifecycle 篩選的字串拼接），需要立即修復。**

## 三、風險摘要與修復優先級

| 優先級 | 問題 | 風險等級 | 修復方式 |
|--------|------|----------|----------|
| **P0 - 立即修復** | `getCustomerRegistrationTrend` 中 lifecycle 參數直接拼接到 sql.raw()，存在 SQL 注入 | **高** | 改用 Drizzle 的參數化查詢或手動跳脫/白名單驗證 |
| **P1 - 盡快修復** | 客戶/訂單 API 返回 rawData 欄位，暴露不必要的完整原始資料 | **中** | 在 select 中排除 rawData 欄位 |
| **P1 - 盡快修復** | list API 的 limit 參數沒有上限 | **中** | 加入 `.max(500)` 限制 |
| **P1 - 盡快修復** | batch import API 沒有檢查 batch 陣列大小 | **中** | 加入陣列長度上限檢查（如 2000 筆） |
| **P2 - 建議改善** | 沒有 Rate Limiting | **中** | 加入 express-rate-limit |
| **P2 - 建議改善** | 沒有安全 HTTP Headers | **中** | 加入 helmet 中間件 |
| **P3 - 長期改善** | export API 一次查詢 100,000 筆 | **低** | 考慮分頁匯出或串流下載 |

## 四、回答原始問題

> 「能不能不要把所有資料都寫在前端尤其是資料庫」

**我們的系統沒有這個問題。** 資料庫連線字串、JWT 密鑰、後端 API Key 等敏感資訊都只存在於後端環境變數中，前端程式碼完全看不到。

> 「沒防大量資料注入可以隨便污染資料庫」

**部分有防護，但可以更好。** 所有寫入操作都需要登入認證和權限檢查，批次操作有 5,000 筆上限。但 batch import API 缺少陣列大小檢查，list API 的 limit 參數沒有上限，也沒有 rate limiting。

> 「沒鎖一些個資的 API，F12 network 點進去就看光光了」

**這是我們需要改善的地方。** 雖然所有 API 都需要登入才能存取（未登入會被擋），但已登入的使用者在 F12 Network 中確實可以看到 API 回傳的完整客戶資料，包括 `rawData` 欄位中的原始匯入資料。我們應該在 API 回傳時排除不必要的欄位。
